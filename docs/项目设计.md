## 1. 项目概述

### 1.1 项目名称与定位

| 项 | 内容 |
|---|---|
| 项目名 | **BizMind**（Business Mind — 企业知识智能助手） |
| 类型 | 多租户 Agentic RAG 平台 |
| 一句话 | 面向企业私有文档的多租户 Agentic RAG 平台，带权限隔离、文档版本感知、检索质量自评与可量化评测。 |

### 1.2 背景与动机

现有 GitHub 仓库（HIS-Go、ERP-Go、WorkPal 等）展示了**复杂业务系统与后端工程**能力，但缺少可公开演示的 **AI 应用**项目。AI 应用开发岗位面试通常考察：

- LLM 集成（流式、Function Calling、成本控制）
- RAG 全链路（Chunk、Embedding、Hybrid Retrieval、Rerank）
- Agent 编排（LangGraph 状态机、条件分支、Tool Use）
- 工程化（多租户、权限、缓存、限流、评测、可观测性）

BizMind 以「企业知识场景」为切入点，与已有 HIS/ERP 领域经验形成叙事闭环，避免「ChatGPT 套壳」标签。

### 1.3 目标用户与场景

| 角色 | 场景 |
|------|------|
| 企业普通员工 | 查询内部 SOP、制度、操作手册 |
| 部门管理员 | 上传/更新本部门文档，查看使用统计 |
| 系统管理员 | 用户管理、全局配置、评测基准维护 |

**Demo 数据集（虚构，自行编写）：**

1. **HIS 风格** — 医院信息系统操作手册（挂号、医嘱、权限）
2. **ERP 风格** — 跨境电商库存/报关/进销存 SOP
3. **WorkPal 风格** — 项目管理规范、迭代流程、缺陷分级

### 1.4 非目标（YAGNI）

以下能力**不在 v1.0 范围内**，避免 scope 膨胀：

- 模型 Fine-tuning / 自研 Embedding 模型
- 多模态（图片/语音/视频）理解
- 自研向量数据库
- 复杂 MCP 生态（可作为 v1.1 可选加分项）
- 与真实 HIS/ERP 生产系统对接

| CI | **GitHub Actions** | — | lint + test + docker build |

### 2.2 关键设计决策（ADR 摘要）

#### ADR-001：LangGraph 而非固定 Pipeline

- **决策：** 使用 LangGraph 构建 Agent 工作流。
- **理由：** 检索质量不确定时需要条件分支（rewrite / web fallback / critique retry）；固定 RAG pipeline 无法表达，面试也难以讲清「决策逻辑」。
- **代价：** 学习曲线略高于 LangChain Chain；节点需单独测试。

#### ADR-002：Qdrant 而非 Chroma

- **决策：** 向量存储选 Qdrant。
- **理由：** payload filter 支持多租户 namespace；hybrid search 原生支持；生产级 Docker 部署成熟。
- **备选：** Milvus（更重）；Chroma（轻量但 multi-tenant filter 较弱）。

#### ADR-003：Parent-Child Chunking

- **决策：** 子块（child）入向量库检索，父块（parent）作为 LLM 上下文。
- **理由：** 小块提高检索精度，大块保证生成上下文完整；企业 SOP 文档结构清晰，适合此策略。
- **参数初值：** child 512 tokens / overlap 64；parent 2048 tokens。

#### ADR-004：文档版本感知 Thread

- **决策：** 每次文档库变更 bump `documents_version`；会话绑定 `(user_id, documents_version)`。
- **理由：** 生产 RAG 常见痛点——文档更新后旧对话引用过期 chunk；体现工程思维。
- **行为：** 版本不一致时，UI 提示「知识库已更新，建议开启新对话」。

#### ADR-005：Baseline RAG 作为对照组

- **决策：** 保留简单单向量 RAG 路径（`/chat/baseline` 或 feature flag）。
- **理由：** 评测与面试可量化 Agent 增益；A/B 对比有说服力。

### 3.2 LangGraph Agent 工作流

```
                    ┌─────────┐
                    │  START  │
                    └────┬────┘
                         │
                    ┌────▼────┐
                    │ Router  │  分类：direct / rag / web / oos
                    └────┬────┘
                         │ rag
                    ┌────▼────────┐
                    │  Retrieve   │  Hybrid + Rerank → top-k chunks
                    └────┬────────┘
                         │
                    ┌────▼────┐
                    │  Grade  │  LLM 评判检索相关性
                    └────┬────┘
           ┌─────────────┼─────────────┐
           │ ok          │ low         │ empty
      ┌────▼────┐  ┌─────▼─────┐ ┌─────▼─────┐
      │ Generate│  │Rewrite Qry│ │Web Search │
      └────┬────┘  └─────┬─────┘ └─────┬─────┘
           │             │             │
           │             └──────┬──────┘
           │                    │
           │               (retry retrieve, max 2)
           │                    │
      ┌────▼────────────────────▼────┐
      │          Generate            │
      └──────────────┬───────────────┘
                     │
                ┌────▼────┐
                │ Critique│  幻觉/遗漏自检
                └────┬────┘
           ┌─────────┼─────────┐
           │ pass    │ fail    │
      ┌────▼────┐ ┌──▼───┐
      │   END   │ │Retry │ (max 1, 带修正 prompt)
      └─────────┘ └──┬───┘
                     └──► Generate
```

**AgentState 核心字段：**

```python
class AgentState(TypedDict):
    messages: list[BaseMessage]       # 对话历史
    query: str                        # 当前用户问题
    rewritten_query: str | None       # 改写后查询
    retrieved_chunks: list[ChunkRef]  # 检索结果
    retrieval_score: float            # 检索质量分
    retrieval_attempts: int           # 重试次数
    generation: str                   # 当前生成答案
    critique_passed: bool
    critique_feedback: str | None
    citations: list[Citation]         # 引用溯源
    route: Literal["direct", "rag", "web", "oos"]
```

### 4.1 模块边界原则

| 原则 | 说明 |
|------|------|
| **api 层只做 HTTP** | 参数校验、鉴权、调用 service，不含业务逻辑 |
| **agent 与 rag 分离** | `rag/` 可独立单测；`agent/` 只编排，不直接操作 ORM |
| **prompts 外置** | 模板放 `agent/prompts/`，便于迭代与版本对比 |
| **配置集中** | 所有 magic number 进 `config.py`（top_k、chunk_size、max_retries） |

### 5.3 Qdrant Collection 设计

- **Collection 名：** `bizmind_chunks`
- **Payload 字段：** `tenant_id`, `document_id`, `parent_id`, `chunk_id`, `chunk_type`（child/parent）, `source`, `page`, `text_preview`

- **向量维度：** 与 Embedding 模型一致（如 1024 for BGE-M3，1536 for text-embedding-3-small）

## 6. API 设计概要

**Base URL：** `/api/v1`  
**认证：** `Authorization: Bearer <JWT>`  
**流式：** `Content-Type: text/event-stream`

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/auth/register` | 注册 |
| POST | `/auth/login` | 登录，返回 JWT |
| GET | `/auth/me` | 当前用户 |
| POST | `/documents/upload` | multipart 上传 PDF/MD |

### 7.2 分支策略

### 7.3 测试策略

| 层级 | 工具 | 覆盖目标 |
|------|------|----------|
| 单元测试 | pytest | chunking、retriever 打分、agent 各 node 纯函数 |
| 集成测试 | pytest + httpx AsyncClient | Auth、上传索引、SSE 端到端（mock LLM） |
| Agent 测试 | pytest + mock LLM responses | 条件边走向（grade low → rewrite） |
| 评测测试 | RAGAS batch | Golden QA 回归，CI 可选 nightly |
| 前端 | Vitest + Testing Library | 关键组件、SSE hook |

**Mock 策略：** 集成测试使用 `pytest-httpx` 或 LangChain `FakeListChatModel`，避免 CI 消耗 API 额度。

**覆盖率目标：** backend 核心模块（`rag/`, `agent/`, `services/`）≥ 70%；不追求 100%。

### 7.7 日志与可观测性

| 维度 | 方案 |
|------|------|
| 结构化日志 | `structlog` JSON 格式；每条带 `request_id`, `user_id`, `tenant_id` |
| 分布式追踪 | Langfuse trace：一次 chat 对应一条 trace，每个 Agent node 一个 span |
| 指标 | 请求延迟、token 用量、检索命中率；v1 写 PG，后续可接 Prometheus |
| 错误处理 | 统一 `AppException` → HTTP 状态码 + 错误码枚举 |

### 7.8 安全设计

| 威胁 | 措施 |
|------|------|
| 跨租户数据泄露 | API + Qdrant filter 双重校验 `tenant_id` |
| JWT 泄露 | 短过期 + refresh token（v1 可仅 access 24h） |
| 文件上传攻击 | 白名单 mime、大小上限（20MB）、病毒扫描（v2） |
| Prompt 注入 | 系统 prompt 隔离用户输入；检索内容用 XML 标签包裹 |
| API 滥用 | Redis 滑动窗口限流（用户级 + IP 级） |
| 密钥管理 | 仅 `.env` / Docker secrets；README 明确禁止 commit |

### 7.9 性能与成本

| 优化点 | 策略 |
|--------|------|
| Embedding 重复计算 | Redis cache：`hash(text) → vector`，TTL 7d |
| LLM 成本 | 路由简单问题走 `gpt-4o-mini`；critique 用更小模型 |
| 检索延迟 | top_k 先 20 再 rerank 到 4；Qdrant HNSW 调参 |
| 流式体验 | 首 token 目标 < 2s；retrieve 与 history summary 并行 |

## 8. 评测体系

### 8.1 Golden QA 集

- 格式：`data/golden_qa.jsonl`，每行 `{question, expected_answer, doc_refs[], category}`
- 数量：≥ 20 条，覆盖 HIS / ERP / WorkPal 三类
- 分类：单跳 factual、多跳综合、拒答（文档外问题）、版本敏感

### 8.2 RAGAS 指标

| 指标 | 含义 | v1 目标 |
|------|------|---------|
| faithfulness | 答案是否忠于上下文 | ≥ 0.85 |
| answer_relevancy | 答案是否切题 | ≥ 0.80 |
| context_precision | 检索是否精准 | ≥ 0.75 |
| context_recall | 检索是否召回关键信息 | ≥ 0.70 |

### 8.3 对比实验

README 中展示表格：

| 模式 | faithfulness | context_precision | avg_latency |
|------|-------------|-------------------|-------------|
| Baseline RAG | 0.72 | 0.61 | 1.2s |
| Agentic RAG | 0.88 | 0.79 | 2.8s |

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| LLM API 不稳定 | Demo 失败 | 配置备用 base_url；集成测试 mock |
| Rerank 模型体积大 | 部署慢 | 可选 Cohere API；Docker 多阶段构建 |
| Scope 膨胀 | 延期 | 严格按 Phase 交付；非目标写死 |
| 评测分数低 | 面试说服力弱 | 优先优化 chunk + rerank，再调 Agent |
| 前端耗时过多 | 后端完不成 | P1 可用简易 UI；P2 再 polish |
